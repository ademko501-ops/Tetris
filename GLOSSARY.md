# GLOSSARY — Словарь терминов

## Модуль 3 · Этап R-1 — Веб-основы

### HTML
**HTML** — HyperText Markup Language, «язык гипертекстовой разметки».
Описывает **структуру** страницы: где заголовок, где блок, где картинка.
Это не программирование, а разметка — как теги на коробках на складе.
*Аналогия из систем безопасности:* план-схема объекта с расстановкой
камер и обозначениями зон — сам по себе ничего не делает,
но говорит, ЧТО и ГДЕ находится.

### CSS
**CSS** — Cascading Style Sheets, «каскадные таблицы стилей».
Описывает **оформление**: цвета, размеры, шрифты, расположение.
HTML отвечает на «что», CSS — на «как это выглядит».
*Аналогия из систем безопасности:* дизайн рабочего места оператора —
как выглядят шкафы с оборудованием, корпуса камер, цвета индикаторов.
Функционал тот же, а вид совсем другой.

### JavaScript (JS)
**JavaScript** (сокращённо **JS**) — язык программирования для браузера.
Отвечает за **поведение**: реакции на клики, движение фигур,
проверки, расчёты. HTML — структура, CSS — вид, JS — действия.
*Аналогия из систем безопасности:* логика VMS, которая по детекции
движения включает запись, по тревоге переключает монитор на нужную
камеру, по расписанию выводит PTZ на пресет. «Мозги» системы.

### Браузер
**Браузер** — программа, которая открывает веб-страницы.
Читает HTML, применяет CSS, запускает JavaScript и показывает результат.
Примеры: Chrome, Edge, Firefox, Safari.
*Аналогия из систем безопасности:* клиент VMS (Operator Client) —
программа, через которую оператор смотрит видеопотоки и управляет
системой. Сами камеры/серверы есть и без него, но видишь их через клиента.

### Консоль браузера (F12)
**Консоль** — встроенная панель отладки в браузере, открывается
клавишей **F12** (или Ctrl+Shift+I). Внутри много вкладок,
главные: **Console** — логи и ошибки JS, **Elements** — текущая
структура страницы, **Network** — какие файлы загрузились.
*Аналогия из систем безопасности:* журнал событий VMS / окно
диагностики — там видно ошибки кодеков, отвалившиеся камеры,
сообщения от устройств. Без него «всё работает или не работает»,
с ним — понятно ПОЧЕМУ.

### CSS Grid
**CSS Grid** — встроенная в CSS система раскладки элементов
по **сетке из строк и колонок**. Задаёшь сколько колонок,
сколько строк, какого размера ячейки — браузер сам расставит.
*Аналогия из систем безопасности:* видеостена из мониторов в
ситуационном центре, разбитая на матрицу 4×4 или 3×3.
Каждая ячейка стандартного размера, картинка попадает в свою
ячейку автоматически.

### Моноширинный шрифт
**Моноширинный** (он же **monospace**) — шрифт, в котором **все символы
одинаковой ширины**. Буква «i» занимает столько же места, сколько «W».
Примеры: Courier New, Consolas, Monaco. Удобен для кода и интерфейсов
«терминального» вида.
*Аналогия из систем безопасности:* цифровая индикация на старых
видеорегистраторах и таймкод поверх записи — цифры не «прыгают»,
потому что каждая занимает фиксированную ширину.

### Фреймворк
**Фреймворк** (framework, «каркас») — готовая большая библиотека/набор
правил, которая берёт на себя кучу рутины: маршрутизацию, отрисовку,
управление состоянием. Примеры для веба: React, Vue, Angular.
Плюсы: быстро. Минусы: своя сложность, надо учить «как принято в нём».
*Аналогия из систем безопасности:* готовая VMS-платформа
(Bosch BVMS, Milestone, Genetec) против написания своего софта
для камер с нуля. Платформа даёт стандартные блоки, но навязывает
свои правила работы.

> **В нашем проекте мы НЕ используем фреймворков** — пишем
> «руками» на чистых HTML + CSS + JavaScript, чтобы понимать
> каждую строчку.

## Модуль 3 · Этап R-2 — Логика и данные

### Тетромино
**Тетромино** (tetromino) — фигура из четырёх квадратов,
соединённых сторонами. Имя от греческого «тетра-» (четыре) + «домино».
В классическом Тетрисе **семь** тетромино: **I, O, T, S, Z, L, J**
(по форме напоминают эти буквы).

### Массив
**Массив** (array) — упорядоченный список значений с доступом
по номеру (индексу). В JavaScript записывается квадратными скобками:
`const arr = [10, 20, 30];` — здесь `arr[0]` это `10`, `arr[1]` это `20`.
Индексация с нуля.
*Аналогия из систем безопасности:* список камер в системе по номерам
(камера №0 — вход, №1 — парковка...). Открываешь по индексу — и сразу
понятно какая.

### Двумерный массив
**Двумерный массив** — массив, элементами которого являются другие
массивы. Обращение через два индекса: `board[row][col]` — сначала
номер строки, потом номер столбца. Удобен для сеток, таблиц и
игровых полей.
*Аналогия из систем безопасности:* видеостена из мониторов —
матрица 4×4. Чтобы указать конкретный монитор, говорим «строка 2,
колонка 3».

### Функция
**Функция** (function) — именованный кусок кода, который можно
**вызвать** сколько угодно раз. Принимает на вход аргументы
(может и без них), внутри что-то делает, может вернуть результат.
В JavaScript:
```js
function drawBoard() { ... }
drawBoard();   // вызов
```
*Аналогия из систем безопасности:* макрос/preset на пульте оператора —
нажал кнопку «Тревога-1», и заранее прописанная цепочка действий
(переключить камеры, начать запись, отправить email) выполнилась.

### drawBoard
**`drawBoard()`** — наша конкретная функция в `src/game.js`.
Читает массив `board` и для каждой ячейки создаёт div-элемент
в HTML: пустой = класс `cell`, занятый = классы `cell filled`.
Это «художник», который превращает данные в картинку на экране.

### diff
**Diff** (от «difference», разница) — построчная разница между
двумя версиями файла. Git показывает diff командой `git diff`:
**красным** — удалённые строки, **зелёным** — добавленные.
На GitHub diff виден на странице любого коммита и Pull Request.
*Аналогия из систем безопасности:* сравнение двух конфигов VMS
до и после изменения — что именно поменялось в настройках камер,
правах пользователей или расписаниях записи.

### Pull Request (PR)
**Pull Request** (PR), он же **Merge Request** в GitLab — предложение
«влить» (merge) изменения из одной ветки в другую на GitHub.
Обычная схема в командной работе: разработчик пилит фичу в своей
ветке, открывает PR в `main`, коллеги ревьюят diff, дают замечания,
после одобрения PR сливается.
*Аналогия из систем безопасности:* заявка на изменение конфигурации
(change request) с приложенным diff'ом — старший инженер проверяет,
что предложено, и одобряет (или отклоняет) перед применением на боевую систему.
> В нашем учебном проекте PR пока не используем — коммитим прямо
> в `main` и пушим. PR нам пригодится, когда появятся отдельные
> ветки под фичи или код-ревью.

## Модуль 3 · Этап R-3 — Анимация и таймеры

### setInterval / clearInterval
**`setInterval(fn, ms)`** — встроенная функция браузера. Запускает
функцию `fn` повторно каждые `ms` миллисекунд. Возвращает «идентификатор»
таймера — число, по которому потом можно его остановить через
**`clearInterval(id)`**.
Пример из нашего кода:
```js
const fallTimer = setInterval(dropStep, 1000); // dropStep раз в секунду
// ...когда фигура достигла дна:
clearInterval(fallTimer);                       // стоп
```
*Аналогия из систем безопасности:* расписание (schedule) в VMS —
каждые N секунд VMS опрашивает камеру, делает снимок, проверяет
соединение. Когда расписание отключают — действия прекращаются.

### tShape
**`tShape`** — переменная в `src/game.js` с **формой фигуры T**
в виде двумерного массива:
```js
const tShape = [
  [1, 1, 1],
  [0, 1, 0],
];
```
`1` — занятая клетка, `0` — пустая. Координаты внутри `tShape` —
это смещения от верхне-левого угла фигуры, **независимые** от её
положения на поле. На следующих этапах рядом появятся `iShape`,
`oShape`, `lShape` и т. д. — все 7 классических тетромино.
*Аналогия из систем безопасности:* шаблон раскладки видеостены
(layout) — описывает «как разложить мониторы относительно друг
друга», но не привязан к конкретному залу. Шаблон один, применить
можно в разных местах.

### pieceRow / pieceCol
**`pieceRow`** и **`pieceCol`** — координаты **верхне-левого угла
падающей фигуры** в системе координат поля. Меняются по мере движения:
`pieceRow++` при шаге вниз, в будущем `pieceCol--` / `pieceCol++` при
движении влево/вправо. Объявлены через `let` (а не `const`), потому
что значения переписываются.
*Аналогия из систем безопасности:* PTZ-координаты камеры
(`pan`, `tilt`) — две числовые переменные, которые описывают
«куда сейчас смотрит» камера. Меняешь значение — камера поворачивается.

### Протокол автоисправления
**Протокол автоисправления** — правило проекта (введено в R-3):
после прогона vault-reviewer Claude Code сам чинит замечания
**низкого** и **среднего** приоритета (отдельным коммитом с
префиксом `fix:`), а на **высоком** останавливается и идёт за
решением к архитектору. Полная формулировка — в `CLAUDE.md`,
раздел «Протокол автоисправления после ревью».

## Модуль 3 · Этап R-4 — Управление клавиатурой

### addEventListener
**`addEventListener('событие', функция)`** — встроенный метод браузера
для регистрации обработчика события. Говорит браузеру: «когда случится
вот это событие — позови вот эту функцию». Можно вешать на разные цели:
на весь документ, на отдельный `<div>`, на поле ввода.
Пример из нашего кода:
```js
document.addEventListener('keydown', (event) => { ... });
```
*Аналогия из систем безопасности:* подписка на алармы в VMS — оператор
настраивает: «когда сработает датчик двери — выведи камеру №3 на
полный экран и начни запись». Система сама позовёт нужный сценарий,
оператору не надо постоянно опрашивать датчик.

### keydown
**`keydown`** — имя события «клавиша нажата». Срабатывает в момент,
когда клавиша уходит вниз (ещё до того, как её отпустили). Соседи
по семейству: **`keyup`** — клавишу отпустили; **`keypress`** —
устаревший вариант, лучше не использовать. Для игр обычно нужен
именно `keydown` — реакция должна быть мгновенной, без ожидания,
что игрок снимет палец.
*Аналогия из систем безопасности:* реакция на «фронт» сигнала
от датчика (момент срабатывания), а не на «спад» (момент возврата
датчика в норму). VMS-логика обычно вешается именно на фронт.

### event.key
**`event.key`** — поле объекта-события `keydown`, в котором лежит
**имя нажатой клавиши** в виде строки. Примеры значений:
- `'a'`, `'b'`, `'1'` — буквы и цифры;
- `'Enter'`, `'Escape'`, `'Tab'`, `' '` (пробел) — спецклавиши;
- `'ArrowLeft'`, `'ArrowRight'`, `'ArrowUp'`, `'ArrowDown'` — стрелки.
Сравниваем через `===`:
```js
if (event.key === 'ArrowLeft') { tryMoveHorizontal(-1); }
```
*Аналогия из систем безопасности:* идентификатор кнопки на физическом
пульте оператора. Каждой кнопке присвоено имя/код, по которому VMS
понимает, что именно нажали, и решает, какой макрос запустить.

## Модуль 3 · Этап R-5 — Таймеры, soft drop и поведение браузера

### keyup
**`keyup`** — парное к `keydown` событие «клавиша отпущена». Срабатывает
в момент, когда игрок снимает палец с клавиши. Нужно там, где важно
не только начало нажатия, но и его конец: удерживание ↓ для ускоренного
падения, удерживание прицеливания в шутерах, удерживание PTT в рации.
```js
document.addEventListener('keyup', (event) => {
  if (event.key === 'ArrowDown') { startFallTimer(FALL_INTERVAL_MS); }
});
```
*Аналогия из систем безопасности:* событие «сигнал тревоги снят»
(возврат датчика в норму). В VMS оно нужно, например, чтобы остановить
запись «по тревоге» или вернуть монитор в обычный режим — то есть
закрыть действие, которое было запущено по `keydown`-собрату.

### event.repeat
**`event.repeat`** — поле объекта-события `keydown`. **`true`**, если
это срабатывание — ОС-автоповтор от удерживаемой клавиши, **`false`** —
если это первое нажатие. Браузер сам не отличает «зажал» от «бьёт по
клавише», он генерирует одинаковые `keydown` несколько раз в секунду.
Чтобы реагировать на удержание только один раз, проверяем `event.repeat`:
```js
if (event.repeat) { return; }
```
*Аналогия из систем безопасности:* в журнале от датчика приходит серия
одинаковых тревог подряд (датчик «трясёт»). VMS-логика часто отличает
«первое событие» от «повтор того же» — чтобы не запускать макрос
многократно, пока сигнал держится.

### clearInterval
**`clearInterval(id)`** — встроенная функция браузера. Останавливает
таймер, который был запущен через `setInterval`. На вход — тот самый
**идентификатор**, который `setInterval` вернул при создании:
```js
const fallTimer = setInterval(dropStep, 1000); // запустили
clearInterval(fallTimer);                       // остановили
```
Парная к `setInterval`, как `keyup` к `keydown`. Если не остановить —
таймер будет бить бесконечно, даже когда логика игры этого уже не ждёт.
*Аналогия из систем безопасности:* отмена расписания опроса камеры
в VMS. Пока расписание активно — VMS дёргает камеру каждые N секунд.
Удалили расписание — дёрганья прекращаются.

### event.preventDefault
**`event.preventDefault()`** — встроенный метод объекта-события. Говорит
браузеру: «не делай своё стандартное действие для этого события».
Примеры стандартных действий, которые надо отключать в играх:
- стрелки ↓ / ↑ скроллят страницу;
- пробел тоже скроллит (на полстраницы вниз);
- правая кнопка мыши открывает контекстное меню;
- F5 перезагружает страницу.
В нашем коде `preventDefault` вызывается в начале `keydown` для трёх
играющих клавиш из `CONTROLLED_KEYS` — иначе во время игры страница
уехала бы под фигурой.
*Аналогия из систем безопасности:* перехват нажатий на горячих
клавишах рабочей станции оператора. Например, Ctrl+Alt+Del в Windows
обычно открывает меню — но в режиме «киоск» VMS перехватывает эту
комбинацию и не даёт ОС обработать её по умолчанию.

### Soft drop / Hard drop
**Soft drop** («мягкий сброс») — режим ускоренного падения фигуры, пока
игрок держит ↓. В нашем R-5 это переключение `setInterval` с обычных
1000 мс на быстрые 100 мс. Игрок отпускает клавишу — возвращается
обычная скорость.

Парный термин — **hard drop** («мгновенный сброс», запланирован на R-8):
по нажатию пробела фигура немедленно «телепортируется» к самой нижней
доступной позиции и фиксируется. Без анимации, без таймера, одним кадром.

Оба термина — из канона классического Тетриса. Игроки используют soft
drop, чтобы аккуратно опустить фигуру в нужное место, а hard drop —
когда позиция уже выбрана и надо побыстрее «уронить».
*Аналогия из систем безопасности:* «плавное» и «мгновенное» перемещение
PTZ-камеры. Soft = камера едет к пресету медленно, оператор видит
переход; hard = прыжок в позицию без промежуточных кадров.

## Модуль 3 · Этап R-6 — Стек, столкновения и спавн

### Стек (stack)
**Стек** в Тетрисе — это «осевшие» блоки, которые уже зафиксировались
на поле и больше не двигаются. В коде хранится в массиве `stack[ROWS][COLS]`:
`0` — пусто, `1` — занято. Новая падающая фигура накладывается **поверх**
стека при отрисовке; когда не может опуститься ниже, её клетки тоже
переходят в `stack` как `1` (это и есть «фиксация»).
Важно: «стек» в Тетрисе не имеет ничего общего со «стеком» (LIFO) в
информатике — название историческое, означает «то, что осело на дне».
*Аналогия из систем безопасности:* архив записи в VMS. Live-видео
(падающая фигура) идёт «сверху» — здесь и сейчас. Когда событие
завершилось, кадр уходит в архив (стек) и хранится там как факт.

### Столкновения (collisions)
**Столкновение** — ситуация, когда движущаяся фигура пытается занять
клетку, которая ей не принадлежит:
- **со стеной** — выход за `0..ROWS-1` или `0..COLS-1`;
- **со стеком** — попадание в клетку, где `stack[r][c] === 1`.
В Тетрисе проверка столкновений делается **до** движения: «можно ли
сдвинуть фигуру вниз/влево/вправо?». Если нет — движение отменяется
(для горизонтали) или фигура фиксируется (для вертикали).
В нашем коде вся логика собрана в одной функции `canMove`.
*Аналогия из систем безопасности:* проверка прав доступа до того,
как пускать человека в зону. Сначала смотрим в базе: разрешён ли
вход в этот сектор в это время. Только если «да» — открываем дверь.
Никогда не открываем сначала, а потом смотрим.

### canMove
**`canMove(piece, dRow, dCol)`** — наша функция в `src/game.js`. Отвечает
на вопрос: «может ли фигура `piece` сместиться от текущей позиции на
`dRow` строк вниз и `dCol` колонок вбок?». Перебирает занятые клетки
формы, считает их координаты после предполагаемого сдвига, проверяет
обе ограничивающие условия (границы поля + пересечение со стеком).
Возвращает `true` / `false`. Чистая функция-предикат: ничего не меняет,
только читает состояние.
Используется в трёх местах:
- `dropStep` — `canMove(tShape, 1, 0)` решает «опускать или фиксировать»;
- `tryMoveHorizontal` — `canMove(tShape, 0, deltaCol)` для стрелок ←/→.
*Аналогия из систем безопасности:* «dry-run» проверки конфига перед
применением. Сначала спрашиваем у системы: «эта конфигурация валидна?»
(никаких изменений ещё нет). Если «да» — применяем. Если «нет» —
оставляем как было, без побочных эффектов.

### freezePiece
**`freezePiece()`** — наша функция в `src/game.js`. Берёт занятые
клетки текущей `tShape` (с координатами `pieceRow / pieceCol`) и
переписывает их в `stack` как `1`. С этого момента эти клетки —
часть стека: они уже не двигаются, новые фигуры на них опираются,
горят зелёным как фон.
Технически — простой двойной цикл с условием `if (tShape[r][c] === 1)`.
Полагается на инвариант: «текущая позиция фигуры валидна» (то есть
`canMove` гарантировала, что мы внутри поля). Самостоятельных
проверок границ не делает.
*Аналогия из систем безопасности:* «сохранить событие в архив».
Live-камера показывала движение в кадре — оператор подтвердил, событие
прошло запись «по тревоге» → кадры переходят в постоянное хранилище
и помечаются как факт инцидента.

### spawnNewPiece
**`spawnNewPiece()`** — наша функция в `src/game.js`. Возвращает
`pieceRow` и `pieceCol` в стартовую позицию: верх поля, центр по
горизонтали. Вызывается дважды:
1. при старте игры — для самой первой фигуры;
2. сразу после `freezePiece()` — для следующей.
В R-6 форма фигуры всегда `tShape` (буква T). В R-7 здесь же будет
случайный выбор из семи тетромино, и `tShape` сменится на переменную
`currentShape`. Формула центрирования живёт ровно в одном месте —
в этой функции.
*Аналогия из систем безопасности:* «вызвать следующую плановую задачу».
Очередь патрулей, цикл смен оператора, последовательность регулярных
проверок камер — каждый новый цикл начинается из одного и того же
стартового состояния.

### event.code
**`event.code`** — поле объекта-события `keydown` / `keyup`, парное
к уже знакомому `event.key`. Разница:
- **`event.key`** — это **символ**, который получится от нажатия с
  учётом раскладки, регистра и модификаторов. Пример: на русской
  раскладке клавиша «Q» вернёт `event.key === 'й'`. Для пробела
  `event.key === ' '` (литерал пробела).
- **`event.code`** — это **физическое имя клавиши**, всегда одно и
  то же независимо от раскладки и модификаторов. Пример: та же клавиша
  «Q» в любой раскладке вернёт `event.code === 'KeyQ'`. Для пробела
  `event.code === 'Space'`.
Для стрелок оба свойства совпадают (`'ArrowLeft'`, `'ArrowRight'` и т. д.),
поэтому при добавлении пробела в R-6 мы перешли на `event.code` во
всём блоке клавиатуры — это даёт читаемое `'Space'` вместо хрупкого
`' '` и сразу делает код layout-агностичным (будущие буквенные хоткеи
работают на любой раскладке).
*Аналогия из систем безопасности:* серийный номер устройства против
его читаемого имени. На пульте «Камера 3» может переехать в «Камера 5»
при перенумерации, но MAC-адрес/серийник останется тем же. Для надёжной
автоматики ссылаемся на серийник (`event.code`), а имя (`event.key`) —
для отображения оператору.

## Модуль 3 · Этап R-7 — Каталог фигур, 7-bag и Fisher–Yates

### SHAPES
**`SHAPES`** — наша константа в `src/game.js`, массив из 7 объектов.
Каждый объект описывает одну фигуру:
```js
{ id: 'T', shape: [[1,1,1],[0,1,0]] }
```
Поле `id` — одна буква, имя в каноне Тетриса (I, O, T, S, Z, L, J).
Поле `shape` — прямоугольная 2D-матрица: `1` — занятая клетка, `0` —
пустая.
**Цвет фигуры в `SHAPES` намеренно НЕ хранится.** Цвет живёт в
CSS-переменной `--color-piece-<id>`, а связь между JS и CSS идёт
через id-букву: JS добавляет к клетке класс `piece-<id>`, CSS этот
класс окрашивает. Один источник истины (single source of truth) —
если в R-12 захочется поменять палитру, правка делается только
в `style.css`, в JS лезть не нужно.
*Аналогия из систем безопасности:* база типов устройств в VMS.
У каждого типа камеры есть код и геометрия (где какие порты), но
внешний вид — иконку, цвет на схеме — VMS подтягивает из отдельной
темы оформления.

### 7-bag (мешок семёрки)
**7-bag** — алгоритм случайной выдачи тетромино из канона
современного Тетриса. Идея: вместо «бросаем кубик каждый раз»
берём мешок, в котором лежит ровно по одной копии каждой из семи
фигур. Игрок видит, как одна за другой выпадают все семь —
случайным порядком. Когда мешок пустеет, делаем новый из тех же
семи фигур и снова мешаем.
Что это даёт: **частоты выровнены**. За любые 7 спавнов выпадет
каждая фигура ровно раз. Игрок не страдает от «не было I-фигуры
20 ходов подряд» — такой ситуации просто не может случиться.
В нашем коде мешок хранится в `let currentBag = []`. Функция
`getNextPiece()` достаёт первую фигуру из мешка; если мешок пуст —
зовёт `createBag()`, который тасует копию `SHAPES` Фишером–Йетсом
(см. ниже).
*Аналогия из систем безопасности:* честное распределение задач
между операторами смены. Вместо «звоним случайному оператору»
используется очередь: каждый получает по одной задаче из текущей
семёрки, пока она не закончится. Потом начинается новая семёрка —
снова каждый получает ровно по одной. Никто не работает в три раза
больше остальных просто из-за неудачи рандома.

### Алгоритм Фишера–Йетса (Fisher–Yates shuffle)
**Алгоритм Фишера–Йетса** — стандартный способ равномерно
перемешать массив. Назван в честь авторов: Рональд Фишер и Фрэнк
Йетс (1938 г., изначально для статистических таблиц). В JS:
```js
for (let i = bag.length - 1; i > 0; i--) {
  const j = Math.floor(Math.random() * (i + 1));
  const tmp = bag[i];
  bag[i] = bag[j];
  bag[j] = tmp;
}
```
Идём с конца к началу. На каждом шаге выбираем случайный индекс
`j` из «ещё не зафиксированной» части массива (`0..i` включительно)
и меняем местами `bag[i]` и `bag[j]`. Время O(n), память — O(1)
(тасуем тот же массив). Распределение перестановок — равномерное:
каждая из n! возможных перестановок выпадает с одинаковой вероятностью.
**Почему не `sort()` со случайным сравнителем?** Это популярная
ошибка: `array.sort(() => Math.random() - 0.5)` выглядит как тасовка,
но даёт неравномерное распределение (некоторые перестановки выпадают
заметно чаще). Фишер–Йетс — правильный путь.
*Аналогия из систем безопасности:* перемешивание pin-кодов
доступа при выдаче временных карт. Чтобы рандом был «честный» и
кодов хватало без коллизий, используют именно алгоритм-перестановку,
а не «случайное число из всего диапазона на каждом шаге».

### createBag / getNextPiece
**`createBag()`** — наша функция в `src/game.js`. Возвращает копию
массива `SHAPES` в случайном порядке (через Фишера–Йетса). Оригинал
`SHAPES` не трогает — копию делает через `.slice()`. Чистая функция:
с одинаковым внутренним рандомом всегда вернёт одно и то же.

**`getNextPiece()`** — её парная функция. Достаёт первую фигуру из
`currentBag` через `.shift()`. Если мешок пуст — сначала пересоздаёт
его через `createBag()`, а потом достаёт. Возвращает один объект из
`SHAPES` (`{id, shape}`).
Эти две функции — фасад для 7-bag-алгоритма. Внешний код (а конкретно
`spawnNewPiece()`) знает только про `getNextPiece()` и ничего не
помнит про мешок: получил фигуру — поставил её на поле.
*Аналогия из систем безопасности:* `createBag` — это «составить
маршрутный лист на смену»; `getNextPiece` — это «дай следующий
объект для обхода». Оператор знает только второе; первое
происходит автоматически в начале смены.

### --spacing-base
**`--spacing-base: 20px`** — наша CSS-переменная в `:root` (`style.css`).
Базовый отступ страницы. Применяется в двух местах:
- `body padding: var(--spacing-base)` — отступ всего контента от
  краёв окна;
- `h1 margin: var(--spacing-base) 0 30px 0` — отступ заголовка
  сверху (одинаковый с padding'ом тела).
Введена по аудиту R-7, чтобы убрать дублирование `20px` в двух
местах. Поменяешь её — поменяется «дыхание» всей страницы сразу.
*Аналогия из систем безопасности:* единый стандарт «безопасного
расстояния от стены» в плане расстановки оборудования. Поменялся
норматив — пересчиталось всё, без правок каждого чертежа.

### --glow-radius-sm
**`--glow-radius-sm: 8px`** — наша CSS-переменная в `:root`
(`style.css`). Радиус мягкого свечения вокруг фигуры и заголовка.
Применяется в:
- 7 правилах `.piece-<X>` (`box-shadow: 0 0 var(--glow-radius-sm) ...`);
- `h1 text-shadow: 0 0 var(--glow-radius-sm) ...`.
Введена в R-7, чтобы не плодить `8px` в семи местах piece-X и заодно
закрыть низкоприоритетное замечание ревью R-6. Парная переменная
для «большого» свечения — `--glow-radius-lg: 20px`, введена в R-8
по аудиту (см. ниже).
*Аналогия из систем безопасности:* единый радиус «зоны
обнаружения» для всех датчиков одного класса. Если регулятор
изменил норматив — переменная меняется в одном месте, и весь
проект подхватывает.

## Модуль 3 · Этап R-8 — Поворот фигур и SRS

### SRS (Super Rotation System)
**SRS** — официальный стандарт поворотов в современных Тетрисах,
введённый компанией The Tetris Company как часть «Tetris Guideline».
Спецификация публично документирована (см. Tetris Wiki → Super Rotation
System). Состоит из двух обязательных частей:

1. **Канонические спавн-ориентации и квадратные «бокс-рамки» вокруг
   каждой фигуры** — `I` живёт в 4×4 квадрате, `O` в 2×2, остальные
   в 3×3. Только так таблица сдвигов (см. ниже) применяется корректно
   без специальных смещений.
2. **Таблица wall kicks** — для каждой пары соседних состояний поворота
   (0→1, 1→2, 2→3, 3→0) есть **пять проверочных сдвигов**. Алгоритм:
   пробуем приземлить повёрнутую фигуру по первой паре `[dCol, dRow]`,
   не получилось — по второй, и так пока не подойдёт или пока не
   кончатся пять. Если все пять не подошли — поворот отменяется.

В нашем коде SRS живёт в трёх местах: `SHAPES` (канонические рамки),
`SRS_KICKS` (таблица), `tryRotate` (алгоритм).
*Аналогия из систем безопасности:* стандарт ONVIF для IP-камер.
Без него каждый производитель определяет PTZ-команды по-своему;
с ним — единый формат, любая камера понимает любого клиента.

### Wall kicks
**Wall kicks** («сдвиги от стены») — компенсирующие пиксельные сдвиги
фигуры во время поворота. Базовая идея: если просто повернуть фигуру
на месте, она часто упрётся в стену поля или в стек. Чтобы поворот
всё-таки работал, SRS пробует слегка «толкнуть» фигуру в сторону
после поворота — на 1 клетку влево / вправо / вверх или на 2, в
определённой последовательности.

Самый эффектный эффект wall kicks — **T-spin**: специальный поворот
T-фигуры в узкую щель, который без kicks был бы невозможен.

В нашем коде проверка делается через `canMove(rotated, dRow, dCol)`:
пробуем поставить повёрнутую матрицу в позицию `(pieceRow + dRow,
pieceCol + dCol)` — если влезает, применяем; если нет, пробуем
следующий сдвиг из таблицы.
*Аналогия из систем безопасности:* «adaptive parking» в PTZ-камерах
с предустановками — оператор задал прескет, но физически камера упёрлась
в препятствие; контроллер пробует несколько ближайших валидных позиций
вокруг прескета и встаёт в первую доступную.

### rotateMatrix
**`rotateMatrix(shape)`** — наша функция в `src/game.js`. Чистая:
принимает 2D-матрицу, возвращает **новую** матрицу, повёрнутую на
90° по часовой стрелке. Исходную не модифицирует.

Формула CW-поворота: `result[c][rows - 1 - r] = shape[r][c]`.
Геометрический смысл: «строка становится колонкой, но в обратном
порядке».

Для прямоугольной матрицы размера `rows × cols` результат имеет
размер `cols × rows` (ширина и высота меняются местами). Для наших
**квадратных** SRS-боксов (2×2, 3×3, 4×4) размер сохраняется — это
критически важное свойство: пока бокс квадратный, его размер при
повороте не «прыгает», и стандартная таблица wall kicks работает.
*Аналогия из систем безопасности:* поворот плана этажа на 90° в CAD —
координатная сетка остаётся той же, только содержимое перенумеровано.

### currentRotation
**`currentRotation`** — наша state-переменная в `src/game.js`. Целое
число 0, 1, 2 или 3:
- `0` — spawn-состояние (как фигура появилась).
- `1` — R (rotated CW one step).
- `2` — 180° (rotated CW twice).
- `3` — L (rotated CW three times = rotated CCW one step).

Сбрасывается в `0` в `spawnNewPiece()` при появлении каждой новой
фигуры. Инкрементируется через `(currentRotation + 1) % 4` при
успешном повороте. Используется **только** как индекс «FROM-состояние»
в таблице `SRS_KICKS` — алгоритм должен знать, из какого состояния
мы поворачиваем, чтобы выбрать правильные пять сдвигов.
*Аналогия из систем безопасности:* состояние смены в журнале (день /
вечер / ночь / пересменок). Внутренняя логика VMS меняется в зависимости
от состояния — например, какие алармы громко звучат, а какие тихо
складываются в журнал.

### tryRotate
**`tryRotate()`** — наша функция в `src/game.js`. Реализация SRS-алгоритма
поворота в пять шагов:

1. Если текущая фигура — **O**, выход без действия. O симметрична, её
   поворот визуально неотличим — стандартный SRS так и делает.
2. Получить кандидата: `rotated = rotateMatrix(currentShape)`.
3. Выбрать кик-таблицу: `SRS_KICKS.I` для I, `SRS_KICKS.JLSTZ` для
   остальных.
4. Из таблицы взять пять пар `[dCol, dRow]` для текущего `currentRotation`.
   Перебирать по очереди. Первый сдвиг, где `canMove(rotated, dRow, dCol)`
   возвращает `true` — применить: `currentShape = rotated`,
   `currentRotation = (currentRotation + 1) % 4`, `pieceCol += dCol`,
   `pieceRow += dRow`, `drawBoard()`, выход.
5. Если ни один сдвиг не подошёл — выход без изменений. Фигура остаётся
   как была.

Важная гарантия: поворот **никогда** не оставляет фигуру в полу-применённом
состоянии. Либо все четыре значения (`currentShape`, `currentRotation`,
`pieceCol`, `pieceRow`) обновлены вместе и поле перерисовано, либо
ничего не изменилось.
*Аналогия из систем безопасности:* транзакция в БД конфигурации.
Все изменения либо применяются вместе, либо все откатываются — никаких
«пол-настройки сохранилось, пол-нет».
