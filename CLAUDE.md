## Палитра тетромино — архитектурное решение
**Это раздел про цвета падающих фигур, не про UI.**

Палитра Pip-Boy (`#15ff00` + `#ffb000` + тёмный фон, см. раздел «Стиль»
выше) применяется только к **UI-слою**: рамка поля, заголовок, грид-сетка,
свечение поля. Сами **семь тетромино** (I, O, T, S, Z, L, J) играют
расширенной палитрой, осознанно выходящей за пределы Pip-Boy: это
сделано ради читаемости (семь различимых цветов на тёмном фоне) и
ради вселенной Fallout / Vault-Tec в целом — а не только её Pip-Boy-сегмента.

Каноническая палитра проекта:

| Фигура | Hex       | Обоснование |
|:------:|:----------|:------------|
| **I**  | `#39FF14` | Неоновый зелёный — Гейгер-зелёный, радиация, светящаяся колба ядер-колы. Самая «длинная» фигура читается ярче всех. |
| **O**  | `#F4C542` | Золотистый жёлтый — крышки от Nuka-Cola (внутриигровая валюта), родственник янтарного акцента Pip-Boy. |
| **T**  | `#1F4E8C` | Глубокий синий — **фирменный Vault-Tec blue**, цвет формы жителей Убежища. Решение архитектора, подтверждённое дважды (R-7 и R-8). |
| **S**  | `#A6FF00` | Лаймовый — оттенок «токсично-зелёного», флакон Rad-X и стимпак. Отличается от I-зелёного, чтобы их не путать в стеке. |
| **Z**  | `#B22222` | Кирпично-красный — **классический Nuka-Cola red**, банка газировки. Решение архитектора (R-8). |
| **J**  | `#B45A2A` | Рыжий-коричневый — оттенок пустоши, ржавчина брони. |
| **L**  | `#C77B3A` | Светлая ржавчина — светлее J для читаемости на тёмном фоне (исходный `#5A3A22` сливался с фоном, заменён по аудиту R-7). |

**Связь JS ↔ CSS:** в `SHAPES` (`src/game.js`) хранится только id-буква.
Сами hex-значения живут в `:root` (`src/style.css`) как `--color-piece-<id>`.
Связь — через CSS-класс `piece-<id>`, который JS навешивает в `drawBoard()`.
Один источник истины для цветов — CSS, не JS.

### Правило для vault-reviewer
**Цвета фигур — это решение архитектора. НЕ флагать как «выходящие
из палитры Pip-Boy».** Палитра тетромино — отдельная подсистема от
UI-палитры; обе официальные. Если возникает вопрос про конкретный
цвет — сверяться с этой таблицей, а не с разделом «Стиль» выше.

Что reviewer'у всё ещё уместно поднимать:
- читаемость цвета на тёмном фоне (например, `#5A3A22` против `#0a0a0a`
  была реальная проблема — слабый контраст, исправили в R-7);
- предложить новую переменную для повторяющихся значений 8px / 20px / 30px;
- любые «голые» hex-значения **вне** `:root` (дублирование палитры).

## Протокол закрытия этапа
Когда Сергей говорит **«закрой этап R-N»** или **«закрой R-N hash=XXX»**:

а) Проверь, появился ли новый файл `docs/Stage_R-N.pdf`.
б) Обнови `PROGRESS.md` — допиши новый раздел в конец файла
   (старые разделы не переписывать).
в) Обнови `GLOSSARY.md`, если в этапе были новые термины.
г) Сделай атомарный коммит документации:
   - `git add .`
   - `git commit -m "docs: закрыт R-N + артефакт"`
   - `git push`
д) Дай короткий рапорт: что обновлено, хеш нового коммита,
   ссылку на коммит на GitHub.

## Правила атомарных коммитов
- **Код этапа R-N** — отдельный коммит с подписью `"R-N: что сделано"`.
- **Документация** (PROGRESS, GLOSSARY, CLAUDE.md и др.) —
  отдельный коммит с подписью `"docs: ..."`.
- **Правка по замечанию ревью** — отдельный коммит с подписью `"fix: ..."`
  (см. ниже «Протокол автоисправления после ревью»).
- **PDF-артефакт** из `docs/` — может идти вместе с docs-коммитом
  (одним коммитом с обновлением PROGRESS).
- **НЕ смешивать** в одном коммите код и доки.

## Протокол автоисправления после ревью
После каждого прогона vault-reviewer (или другого агента-ревьюера)
Claude Code смотрит на приоритеты найденных замечаний:

- **Низкий / средний приоритет** → Claude Code исправляет сразу,
  без вопросов архитектору. Один проход ревью → одна тематическая
  правка → один коммит. Если замечаний несколько и они близки
  по теме — допускается одним коммитом, разные темы — разными.
- **Высокий приоритет** → Claude Code останавливается, показывает
  замечание Сергею и ждёт его решения. До «ок» не правит и не коммитит.

**Сообщение коммита** — префикс `fix:`. Примеры:
- `fix: rgba комментарий в style.css (замечание vault-reviewer R-3)`
- `fix: убраны лишние console.log по аудиту R-4`

Правки `fix:` — это **отдельный атомарный коммит**: не смешиваются
ни с кодом этапа (`R-N: ...`), ни с документацией (`docs: ...`).
После `fix:`-коммита Claude Code сразу пушит на `main`.

## Папка docs/
Сюда складываются все PDF-артефакты курса.

**Именование:**
- `Stage_R-N.pdf` — этапы кода (например, `Stage_R-1.pdf`, `Stage_R-2.pdf`).
- `Module_N.pdf` или `Module_N-M.pdf` — учебные модули
  (например, `Tetris_Course_Module1.pdf`, `Tetris_Course_Module_2-1.pdf`).

Сергей вручную скачивает PDF из Claude Chat и кладёт в эту папку.
Claude Code эти файлы не генерирует, только коммитит и пушит вместе
с docs-коммитом по протоколу закрытия этапа.
